## Why

The app has run on a single working LLM path since inception: GitHub Models' OpenAI-compatible endpoint (`gpt-4o-mini`, paid for by a GitHub token, not billed). Anthropic support has existed only as an unused config stub. A real Anthropic Console API key is now live on Heroku, and Claude Sonnet 5 should become the new default model — but the GitHub Models path must keep working so existing behavior and cost characteristics aren't silently lost. Today the only way to pick a model at all is a `<select>` that already exists in the new-chat form markup but is hidden (`class="... hidden"`) and unscoped — it lists every provider RubyLLM knows about (~1350 models across gemini, deepseek, mistral, perplexity, xai, azure, bedrock, vertexai, openrouter, ollama, gpustack, in addition to openai and anthropic), almost none of which are configured, so unhiding it as-is would let a user pick a model that immediately raises `RubyLLM::ConfigurationError`.

## What Changes

- Set `RubyLLM.config.default_model` to `claude-sonnet-5` (anthropic provider). **BREAKING**: new chats with no explicit model selection now default to a different provider and model than before.
- Keep `gpt-4o-mini` (openai / GitHub Models) fully configured and selectable — it is no longer the default but remains a first-class option.
- Refresh the app's `models` registry (`Model.refresh!`) so `claude-sonnet-5` exists as a real row (pricing, context window, capabilities) rather than only resolving via `assume_model_exists`.
- Scope `available_chat_models` (`app/controllers/application_controller.rb`) to an **exact `[provider, model_id]` allowlist** (`CHAT_MODELS`), not a provider-level allowlist — a provider allowlist alone turned out to let two kinds of broken model through: non-chat models RubyLLM's registry mistags as `type: "chat"` (`dall-e-3`, `whisper-1`, `tts-1`, `sora-2`, etc.), and same-named models under the unconfigured `azure` provider (`gpt-4o`/`gpt-4o-mini` exist there too). The exact allowlist can't match either.
- Expand the allowlist beyond `claude-sonnet-5`/`gpt-4o-mini` to include `gpt-4o`, `gpt-4.1` (OpenAI, via GitHub Models), `mistral-ai/mistral-small-2503` (Mistral), and `deepseek/deepseek-v3-0324` (DeepSeek) — GitHub Models is a multi-vendor aggregator, not just an OpenAI proxy, and all four were verified live against `models.github.ai` with the current `GITHUB_TOKEN` before being added. The GPT-5 family and the o1/o3/o4 reasoning tier were tested too and are deliberately excluded — they return `400`/`403` on this token/tier.
- Un-hide and style the existing model `<select>` in `app/views/chats/_form.html.erb` per `DESIGN.md` conventions, so a user can choose the model when starting a **new** chat.
- Correct the displayed provider name for the Mistral/DeepSeek entries: they're registered under RubyLLM's `openai` provider slug purely for credential routing (GitHub Models is the only credential we have for them), which made `model.label` show "OpenAI - Mistral Small 3.1" — factually wrong. New `chat_model_provider_label`/`chat_model_label` helpers on `ApplicationController` read the real publisher back from `metadata[:real_publisher]` and are used everywhere a model's provider is displayed (the switcher and the `/models` registry pages).
- Out of scope: switching a model on an already-started chat (`chats/show.html.erb` is unchanged), and surfacing per-chat/per-message cost (`chat.cost` stays unused).

## Capabilities

### New Capabilities
- `chat-model-selection`: Lets a user choose which of an exact, verified set of LLM models powers a new chat at creation time, defaulting to Claude Sonnet 5 when no explicit choice is made. The set spans two credential paths — Anthropic direct, and GitHub Models (which itself proxies multiple publishers: OpenAI, Mistral, DeepSeek) — not just "openai or anthropic providers."

### Modified Capabilities
(none — `chat`, `chat-generation`, and `chat-refinement` specs describe chat ownership, context injection, and generation/refinement flows; none of their existing requirements reference model selection, so this is additive only)

## Impact

- **Config**: `config/initializers/ruby_llm.rb` — `default_model` changes; the existing GitHub-Models-specific payload patch (prefixes outgoing model id with `openai/` only when `openai_api_base` points at `models.github.ai`, and passes through ids that already contain a `/` — which is what makes the `mistral-ai/...`/`deepseek/...` ids work unmodified) is unrelated and untouched.
- **Data**: `models` table gains a `claude-sonnet-5` / `anthropic` row via `Model.refresh!` against the now-live Anthropic API, plus two hand-created rows for `mistral-ai/mistral-small-2503` and `deepseek/deepseek-v3-0324` (both registered under provider `openai`, tagged with `metadata[:real_publisher]`) since GitHub Models' catalog schema isn't something `Model.refresh!` can parse automatically. The committed `config/ruby_llm_models.json` fallback registry is regenerated to match every time the allowlist changes.
- **Code**: `app/controllers/application_controller.rb` (`CHAT_MODELS` exact allowlist, `chat_model_provider_label`/`chat_model_label` helpers), `app/views/chats/_form.html.erb` (un-hide + style the select, use the new label helper), `app/views/models/_model.html.erb` and `app/views/models/show.html.erb` (same label fix — they had the identical mislabeling bug), `app/controllers/chats_controller.rb` (`#new`/`#create` already thread `model` through — confirmed still correct against the narrower list).
- **Cost**: Anthropic's Console API is metered/billed, unlike the free GitHub Models path — this is the first time the app defaults to a paid provider.
- **Operational note**: RubyLLM caches its model registry in memory for the life of the process. Any environment (dev server, production dynos) running before new `Model` rows are created won't see them until restarted — this bit local verification directly (`bin/dev` had to be restarted) and needs the same care in production after `Model.refresh!` or a new deploy that seeds `Model` rows.
- **No changes** to `chats/show.html.erb`, `ChatResponseJob`, `GenerationJob`, or `StructuredExtraction` — they already resolve the model generically through the chat's persisted association.
