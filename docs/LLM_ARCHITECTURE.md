# LLM Architecture — Context, Directives & Delivery Paths

> Design doc for how ContentFlow assembles and delivers instructions to the LLM. Written
> **before** any refactor so the concept survives independently of the code. No app code
> changes are implied by writing this — it is the target we migrate *toward*.
>
> Companion to `USER_JOURNEYS.md` (the chat-driven generation flow) and the inline docs in
> `app/services/llm_context.rb`. Grounded in `ruby_llm` capabilities verified against
> <https://rubyllm.com/> (chat, schemas, tools, agents) in June 2026.

---

## 1. The problem with today's design

Everything the model is told — the assistant's role, the creator's brand facts, the
platform "how-to" guidelines, and the creator's own `custom_instructions` — is concatenated
by `LlmContext#build` into **one** system message and handed to `chat.with_instructions(...)`.

```
preamble → CREATOR PROFILE → PARENT IDEA → PARENT SCRIPT (+ SCRIPT INSTRUCTIONS) → [leaf] → GUIDELINES
```

There is exactly **one delivery path** (a single system string) and **one hardcoded order**
(call-order inside `build`). That makes three things awkward:

- **Paths can't be separated.** Facts and directives ride the same channel; the creator's
  `custom_instructions` is even fused *into* a facts block (`Script#system_prompt` appends it).
- **Ordering isn't flexible.** Precedence is an accident of concatenation order — the platform
  guidelines land last and so implicitly outrank the creator's voice, with no way to declare otherwise.
- **Optional/large context can't scale.** Everything is *pushed* every turn; there's no way to let
  the model *pull* a fact only when it needs it.

---

## 2. The grid: two axes, four kinds of contribution

Every contribution to a prompt is generated by two independent axes:

1. **Facts vs. Directives** — telling the model *what it's working with* (context) vs. *how to behave* (rules).
2. **Developer-owned vs. Creator-owned** — lives in **code** (identical for every account) vs. in **data** (a DB row, specific to this creator/record).

|                | **Developer-owned** (code)                          | **Creator-owned** (data)                                   |
|----------------|-----------------------------------------------------|------------------------------------------------------------|
| **Facts**      | *(rare — folded into the role preamble)*            | **Model `#system_prompt`** — creator profile, idea/script/post fields |
| **Directives** | **Preamble** (global role) + **guidelines** (per-purpose) | **`custom_instructions`** — per-record / per-account steering |

Today's three mechanisms are three of these cells; the preamble is the fourth (developer
directive, always-on). The redesign keeps the grid but gives **each cell its own owner and its
own delivery path**.

---

## 3. RubyLLM channels (the separated delivery paths)

RubyLLM exposes four wire-level channels. The grid maps onto them almost one-to-one — this is
the whole point of the redesign.

| Grid cell | What it is | RubyLLM channel | Rails home |
|---|---|---|---|
| Developer directives | role (preamble) + per-purpose "how-to" | **Agent `instructions`** | `app/agents/*_agent.rb` |
| Output structure | the `title`/`hook`/`body` shape — a *structural* directive | **`schema`** → `response_format` (not prose) | `app/schemas/` *(exists)* |
| Creator facts — small & always-relevant | creator profile, immediate ancestry | *pushed* into `instructions`, sourced from model `#system_prompt` | model methods + agent |
| Creator facts — large or optional | recent posts, Substack inspiration, lookups | **tools** — model *pulls* on demand | `app/tools/*_tool.rb` |
| Creator directives | `custom_instructions` (per record/account) | its **own labeled segment**, distinct from facts | columns + composer |

**Key insight:** the `schema` is already a separated directive path you use today — structure is
delivered as `response_format`, never as English in the prompt. That discipline is the model for
the other cells.

### Verified RubyLLM capabilities this relies on

- **Schemas** — `RubyLLM::Schema` subclasses; attached with `with_schema` or an Agent's `schema` macro. *(Already used by `StructuredExtraction`.)*
- **Tools** — `RubyLLM::Tool` subclasses with `desc`, `param`/`params`, and an `execute` method. Support **constructor injection** (stateful tools: `Tool.new(user: ...)`). Attached with `with_tool(s)`. The model decides when to call them; RubyLLM resolves the call **within a single `ask`** (decide → execute → answer). Good enough to *fetch a fact*; not a heavy multi-step planner.
- **Agents** — `RubyLLM::Agent` subclasses: a reusable, declarative assistant. Macros: `model`, `instructions`, `tools`, `schema`, `temperature`, `params`, `inputs`, `context`, **`chat_model`**. With `chat_model Chat`, the agent persists/streams through an `acts_as_chat` model (`Agent.create!(...)`, `Agent.find(id)`) — i.e. it slots into our existing `Chat` + `ChatResponseJob` flow. `instructions { ... }` and `tools { ... }` accept **lazy blocks** that read `inputs` at runtime — the native way to inject per-creator/per-record state.

## 4. Target architecture

### 4.1 Agent per purpose (the composition root)

One `RubyLLM::Agent` subclass per `purpose` enum value. This is convention-over-configuration:
the string `"generate_linkedin_post"` resolves to `GenerateLinkedinPostAgent`. The agent absorbs
everything developer-owned and purpose-specific — the **preamble + guidelines + schema + model** —
that is currently scattered across `LlmContext` and `GenerationPlan`.

```ruby
# app/agents/generate_linkedin_post_agent.rb   — CONCEPTUAL, not implemented
class GenerateLinkedinPostAgent < RubyLLM::Agent
  chat_model Chat                 # Rails-mode: persists + streams via acts_as_chat
  schema     LinkedinPostSchema
  inputs     :chattable           # the Script being generated from (runtime state)

  # Developer directives (role + how-to), composed in a declared order — see §4.3
  instructions { SystemPrompt.compose(chattable:, purpose: :generate_linkedin_post) }

  # Large/optional facts as PULL, not push — see §4.2
  tools { [ RecentPostsTool.new(user: chattable.user),
            SubstackInspirationTool.new(user: chattable.user) ] }
end
```

### 4.2 Facts: push vs. pull (the decision rule)

The facts row of the grid **splits in two**, and choosing the split is the main new lever tools unlock:

- **Push** (into `instructions`) when the fact is **small and always relevant** — creator profile and
  immediate ancestry. Cheap, deterministic, always present. Source: the existing model `#system_prompt`
  methods. **Keep them.**
- **Pull** (as a **tool**) when the fact is **large, optional, or expensive** — "my last 10 posts,"
  "search my Substack feed for inspiration," "what's trending." Pushing these every turn wastes tokens
  and dilutes attention; a tool lets the model fetch only when the conversation calls for it.

### 4.3 Ordering: segments as data, not concatenation

Replace the hardcoded `join` with an **ordered collection of tagged segments** that a composer sorts.

```ruby
# CONCEPTUAL
Segment = Data.define(:key, :role, :priority, :text)
#   role:     :identity | :facts | :creator_directive | :format_directive
#   priority: decides order; sensible defaults per role, overridable per purpose/agent
```

Each provider contributes `Segment`s; the composer sorts by `priority` and renders into the
agent's single `instructions` string. Benefits:

- **Flexible precedence** — order is data/declaration, not call-order. You *choose* whether the
  creator's voice (`:creator_directive`) outranks the platform format (`:format_directive`), e.g.
  "creator voice wins, except hard constraints like Twitter's 280 chars."
- **Testable without an LLM** — assert segment presence and order directly.

`LlmContext` does not disappear; it **shrinks** into this `SystemPrompt` composer, responsible only
for the facts + directive *prose* (the rows that still belong in the system message). The
purpose-switch and the schema slot move out into the agents.

### 4.4 The separation rules (what "separate paths" means concretely)

1. **One owner per cell.** Facts ← model `#system_prompt`. Developer directives ← agent. Structure
   ← schema. Optional facts ← tools. Nothing concatenates across owners except the final composer.
2. **Un-fuse `custom_instructions` from facts.** Today `Script#system_prompt` appends the creator's
   `custom_instructions` into the same block as the `PARENT SCRIPT` facts — the one place two cells
   share a path. The composer emits them as **separate, labeled segments** so they can be reordered,
   weighted, or routed differently.

## 5. Mapping onto what exists today

| Existing | Fate |
|---|---|
| `Chat` + `acts_as_chat` + `purpose` enum | **Keep.** `purpose` becomes the dispatch key → agent class. |
| `LlmContext` (`app/services/llm_context.rb`) | Shrinks to a `SystemPrompt` **composer** (facts + directive prose, segment-ordered). |
| `preamble` + `platform_guidelines_layer` | Move into each **Agent's** `instructions`. |
| `*Schema` (`app/schemas/`) | **Keep**, attached via the agent's `schema` macro. |
| `StructuredExtraction` (`app/services/structured_extraction.rb`) | Largely **subsumed** — an agent with a `schema` does the structured `ask`. Keep the prompt-JSON fence-strip **fallback** as a safety net. |
| `GenerationPlan` (`app/services/generation_plan.rb`) | Keeps the **Rails side** (owner resolution, persist, redirect); hands schema/instructions to the agent. |
| Model `#system_prompt` (creator/idea/script/posts) | **Keep** as the small-facts (push) source. |
| `ChatResponseJob` + Turbo streaming | **Keep.** `chat_model Chat` agents persist/stream the same way. |

## 6. Phased migration (each phase ships independently, suite stays green)

1. **Segment composer.** Refactor `LlmContext#build` to assemble `Segment`s and sort by priority —
   no behavior change yet (same text, same order via default priorities). Locks in separation +
   testable ordering. Pure refactor.
2. **Un-fuse `custom_instructions`.** Emit it as its own `:creator_directive` segment instead of
   inside `Script#system_prompt`. Decide its precedence explicitly.
3. **Agent per purpose.** Introduce `app/agents/*_agent.rb`, one per `purpose`; move preamble +
   guidelines + schema + model into them; resolve the agent from `chat.purpose`. `LlmContext`
   becomes the composer the agents call. `StructuredExtraction` becomes the fallback-only helper.
4. **First tool.** Add `app/tools/substack_inspiration_tool.rb` (the obvious large/optional fact)
   and attach it to the relevant agents via the `tools { }` block. Proves the pull path end-to-end.

**Recommendation:** stop after as many phases as deliver value. The current ancestry is tiny and
fixed, so **push covers ~90%** of it — phases 1–3 give you separated paths and flexible ordering;
reserve phase 4 (tools/pull) for the first genuinely optional/large fact.

## 7. Open decisions (resolve before/while implementing)

- **Precedence policy.** When the creator's voice and a platform format conflict, who wins, and are
  there non-negotiable "hard constraint" directives (Twitter 280) that always win? (Sets default
  `priority` values in §4.3.)
- **Creator-level directives.** Only `Script` has `custom_instructions` today. Add a
  `Creator#custom_instructions` "house style" that applies across all flows? Per-platform creator
  preferences? Same column pattern, more rows of the bottom-right cell.
- **Guidelines: code vs. data.** Keep developer directives hardcoded (simple, versioned) or make them
  editable by an admin/creator (moves a developer cell into a creator cell — a migration like
  `custom_instructions`)?
- **Rename.** `platform_guidelines_layer` is keyed off *purpose*, not platforms (ideas/scripts aren't
  platforms). Rename to `generation_guidelines` / route through the agent's `instructions`.
- **Refine flow.** Not wired yet. When it is, the chattable becomes the post itself, so its
  `#system_prompt` (facts) runs *and* the agent supplies the directives — confirm the same agent
  serves both generate and refine, or split into `refine_*` agents.

## 8. Verification (when implemented)

- **Composer unit tests** — assert segment set + order per purpose, no LLM call (mirrors the current
  `test/services/llm_context_test.rb` heading/order assertions).
- **Agent resolution test** — each `purpose` resolves to its agent; unknown purpose handled like the
  current `GenerationPlan::UnknownPurpose`.
- **Schema round-trip** — agent with `schema` returns a parsed Hash on the configured endpoint;
  prompt-JSON fallback still covers the endpoint-ignores-`response_format` case.
- **Tool path** — model calls the tool when relevant and ignores it otherwise; tool is user-scoped
  (a tool can only read the signing-in creator's data).
- **End-to-end** — first-run generate flow per platform still produces a saved record in the right
  schema shape and streams in the right register.
